Charge via PayPal without separate Authorization

The integration service allows to capture money from a PayPal account in a one step process without initiating a separate authorization before.

It will do so by sending an authorization request to the PAYONE platform.
It will store information about such a request as an interface interaction of type PAYONE_INTERACTION_REQUEST.

Common Setup

  • payment.paymentMethodInfo.method: WALLET-PAYPAL
  • payment.transactions[0].type: CHARGE

The failed payment Payment 2 must have received 0 responses of type PAYONE_INTERACTION_REDIRECT and its custom field redirectUrl must be null.

If I follow the PayPal redirect process for Payment 1, Payment 3, Payment 4 and verify them successfully with this account, I get forwarded to a predefined success URL and for each of the payments the following notifications will be received:

  1. (txaction=appointed, transactionstatus=completed) received -> has no effect, just as a confirmation
  2. (txaction=paid, transactionstatus=completed or null) received -> The state of the Charge transaction, the authorized amount and the paid amount should be set correctly not later than now.

Note

Due to a discrepancy between the example in the PAYONE Server API documentation and what can be observed by running this executable specification the values in the columns "Transaction State" and "total # of
appointed/completed notifications" aren't asserted, just printed into the table.

In section 4.2.8 Sample: preauthorization, WLT (with “pending”) the PAYONE Server API states that a PAYONE authorization request will lead to appointed/pending, appointed/completed and paid notifications all with parameter sequencenumber=0.

As one can see in the interfaceInteractions list of the payments used in this specification run there are no notifications with sequencenumber=0. Instead there are notifications with sequencenumber=1.

Corresponding to the Integration Service's field mapping specification a new transaction is created in case of a notification with a sequencenumber that doesn't match any of the payment's transactions' interactionId.

The original transactions that triggered the requests to PAYONE remain in state Pending because the notifications cannot be linked to them by the Integration Service.

Further Details

What happens if the verification fails?

That's out of the Integration Service's scope.
The Integration Service is notified by PAYONE only in case of a successful verification.

The checkout implementation needs to take care of failed verifications - the user will be redirected to the backUrl or the errorUrl specified in the Payment object.
